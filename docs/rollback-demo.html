Standard Rollback Workflow Full staging validation before production

🔍 Identify Releases

Look up the current and previous release versions from the release manifest.

# Read manifest
cat releases/2.3.0.json
# Current: 2.3.0 → Rollback target: 2.2.0

🌐 Check Cross-Repo Scope

Determine which repositories were included in the release and need to be rolled back together.

🔎 Staging Contention Check

Verify staging is available. If staging/active exists, wait or poll until it clears.

git fetch origin
git tag -l "staging/active"

🔒 Claim Staging

Tag staging as claimed for the rollback operation.

git tag -a staging/active -m "Claimed for rollback to 2.2.0"

💥 Frontend Cache-Bust Branch

Bump package.json version to force new asset URLs so browsers fetch fresh files.

# package.json version bump
2.2.0 → 2.2.1 (patch bump for cache-bust)

🔃 Reset Staging to Rollback Target

Hard-reset the staging branch to match the rollback target version.

git reset --hard v2.2.0
git push origin staging --force

👁 Validate on Staging Human Gate

Manual validation on the staging environment. Confirm the rollback target behaves as expected before touching production.

🔄 Cross-Repo Version Parity

Verify all related repositories are aligned at the correct rollback version.

🚀 Reset Production (Backend First)

Reset the production branch to the rollback target. Backend repos go first to avoid frontend pointing at non-existent API endpoints.

git reset --hard v2.2.0
git push origin main --force

📝 Annotate GitHub Release

Mark the rolled-back release as rolled back, noting the target version and reason.

🚦 Production Smoke Test Human Gate

Verify production is working correctly after the rollback. Check critical paths and monitoring dashboards.

✅ Post-Rollback Checklist

Release staging claim, notify the team, document the incident, and plan the fix-forward.

Emergency Rollback Workflow Skip staging — for active outages only

⚠️ CONFIRM: This Is an Emergency Human Gate

Explicit acknowledgment that this is an active outage and you are intentionally skipping staging validation. This cannot be undone easily.

WARNING: Emergency mode skips staging validation.
Only proceed if production is actively broken.

🎯 CONFIRM: Target Version Human Gate

Explicitly confirm the version you are rolling back to. No ambiguity allowed in emergency mode.

# Confirm rollback target
Rolling back to: v2.2.0
# Type the version to confirm

💥 Frontend Cache-Bust Branch

Bump package.json version to force fresh asset URLs, even in emergency.

🚨 Reset Production Directly Staging Skipped

Hard-reset the production branch to the rollback target without staging validation. This is the dangerous step that emergency mode enables.

# STAGING VALIDATION SKIPPED
git reset --hard v2.2.0
git push origin main --force

🚦 CONFIRM: CodePipeline Status Human Gate

Check that CodePipeline has picked up the change and is deploying the rolled-back version. Do not proceed until the pipeline is green.

👁 Mandatory Post-Rollback Validation Human Gate

Since staging was skipped, production validation is mandatory and extra thorough. Check all critical paths.

📝 Annotate GitHub Release

Mark the release as emergency-rolled-back with timestamp and reason.

✅ Post-Rollback Checklist

Notify the team, document the incident, schedule a post-mortem, and plan the fix-forward. Extra scrutiny due to emergency mode.

Cache-Busting Explainer

Why?

When rolling back, the code goes backwards but browsers have cached the old assets. Users will see broken pages with stale JS/CSS files.

How?

Bump package.json version to generate new asset URLs. Browsers see unfamiliar filenames and fetch fresh files.

Before & After

app.v2.3.0.js
❌ Cached — stale
➔
app.v2.3.1.js
✅ Fresh — correct
// package.json
"version": "2.2.0" → "2.2.1"
// Patch bump forces new asset filenames
